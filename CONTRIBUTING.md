# Contributing to ForgePath ⚒️

Thanks for your interest in contributing to **ForgePath**!

ForgePath is an open-source starter kit for building an AI-ready Internal Developer Platform. The project is still early, so feedback, ideas, issues, and small improvements are very welcome.

---

## 🚧 Project status

ForgePath is currently a **work in progress**.

Some parts of the project may change quickly.

The goal is to keep the project practical, understandable, and useful as a starter kit.

---

## ✨ Ways to contribute

You can help by:

- improving documentation;
- reporting bugs;
- clarifying setup steps;
- improving the local development experience;
- improving Backstage templates;
- improving GitOps examples;
- improving observability examples;
- improving the AI Incident Analyzer;
- adding masking/redaction patterns.

Small contributions are welcome.

---

## 🧭 Before opening a Pull Request

Please check that your change fits the current scope of ForgePath.

ForgePath is **not** meant to become:

- a production-ready platform;
- a complete AIOps tool;
- an autonomous remediation agent;
- a large template marketplace.

When in doubt, open an issue first.

---

## 🔀 Development workflow

1. Fork the repository.
2. Create a feature branch.
3. Make your changes.
4. Test locally when possible.
5. Update the documentation if needed.
6. Open a Pull Request.

Example branch names:

```text
docs/improve-readme
feat/add-api-template
fix/argocd-bootstrap
chore/update-scripts
```

---

## 📝 Commit style

There is no strict convention yet, but clear commits are appreciated.

Examples:

```text
docs: improve local setup
feat: add Backstage API service template
fix: update ArgoCD application path
chore: clean unused manifests
```

---

## ✅ Pull Request guidelines

A good Pull Request should include:

- a clear title;
- a short explanation of the change;
- validation steps if possible;
- documentation updates if needed.

Please avoid very large PRs. Smaller changes are easier to review.

---

## 🛡️ Security

Do not include real secrets, credentials, tokens, customer data, or production logs in issues, PRs, examples, or tests.

Use fake or synthetic data only.

---

## 🤝 Code of conduct

Be respectful, constructive, and clear.

Disagreements are welcome. Personal attacks are not.
